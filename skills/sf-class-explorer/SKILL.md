# SF Class Explorer — Orchestrator

You run in a **forked context** — isolated from the user's conversation.
The user invoked you with: **$ARGUMENTS** (the component name to analyze).

## Step 1: Detect component type

Determine what $ARGUMENTS refers to:
- **Apex class**: find `$ARGUMENTS.cls` under `*/classes/*`
- **Apex trigger**: find `$ARGUMENTS.trigger` under `*/triggers/*`
- **Flow**: find `$ARGUMENTS.flow-meta.xml` under `*/flows/*`

```bash
find . -name "$ARGUMENTS.cls" -o -name "$ARGUMENTS.trigger" -o -name "$ARGUMENTS.flow-meta.xml" 2>/dev/null
```

If not found, try a partial match and ask the user to confirm.

## Step 2: Launch agents in parallel

Spawn both agents at the same time using Task():

**Agent A — Content analysis:**
- If Apex class or trigger → **sf-source-analyzer** with file path
- If Flow → **sf-flow-scanner** with Flow name and object (extract from XML filename or content)

**Agent B — sf-dependency-scanner** with component name and type

Wait for both to return.

## Step 3: Build the document

Compose a Markdown document with this structure:

---

```markdown
# [ComponentName] — [type: Apex Class / Apex Trigger / Flow]

> Generated on [date] by sf-class-explorer

---

## Functional Overview

### What it does
[2-4 sentences in plain business language. No code references.
Explain the business purpose and when this component is involved.]

### When it runs
[Triggered by / Called from / Launched when — in business terms]

### Key scenarios
- [Scenario 1: when X happens, this does Y]
- [Scenario 2: ...]

---

## Technical Reference

### Logic summary
[From sf-source-analyzer or sf-flow-scanner output — key behaviors, branching conditions]

### Direct dependencies (calls)
[From source/flow analysis]

| Component | Type | Purpose |
|-----------|------|---------|
| ...       | ...  | ...     |

### Inverse dependencies (called by)
[From sf-dependency-scanner]

| Component | Type | Usage |
|-----------|------|-------|
| ...       | ...  | ...   |

### Data
- **Objects read**: [list]
- **Objects written**: [list]
- **Fields of interest**: [list]

### Async / Integrations
[Queueable, Future, Batch, Callouts if any — or "None"]

---

*Generated by sf-class-explorer — sf-ai-skills*
```

---

## Step 4: Save the document

Create `docs/` folder if it doesn't exist.
Save the document as `docs/[ComponentName].md`.

## Step 5: Generate .docx (optional)

Check if Pandoc is available:
```bash
pandoc --version 2>/dev/null | head -1
```

If available: `pandoc docs/[ComponentName].md -o docs/[ComponentName].docx`
If not: mention in the final message that only `.md` was saved.

## Step 6: Deliver

Return to the user:
- Brief summary of what was found (2-3 sentences)
- Paths of files saved
- Any notable dependencies or risks worth highlighting
